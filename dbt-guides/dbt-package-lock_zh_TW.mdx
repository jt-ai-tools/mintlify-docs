當您在 dbt 專案中執行 `dbt deps` 時，您可能會注意到專案目錄中出現了一個新檔案：`package-lock.yml`。此檔案由 dbt-core 自動生成，在確保您的 dbt 專案在不同環境和團隊成員之間一致運行方面發揮著至關重要的作用。

## 什麼是 package-lock.yml？

`package-lock.yml` 是 dbt 的依賴鎖定檔案，當您執行 `dbt deps` 安裝專案依賴項時會自動建立。此檔案記錄了專案中安裝的所有 dbt 套件的準確版本，在特定時間點建立依賴樹的「快照」。

您可以將其視為 dbt 套件的詳細收據——它不僅列出了您購買的內容，還包括準確的版本、來源以及獲取時間。

## 了解套件版本控制 (Package Versioning)

在深入了解鎖定檔案之前，了解 dbt 中套件版本控制的工作原理會很有幫助。

### 語義化版本控制 (Semantic Versioning)

大多數 dbt 套件遵循語義化版本控制 (semver)，使用三部分版本號：`主版本號.次版本號.修補版本號`（例如：`1.2.3`）。

- **主版本號 (MAJOR)**：當有不相容的 API 更改時增加
- **次版本號 (MINOR)**：當有以向下相容方式新增的功能時增加
- **修補版本號 (PATCH)**：當有向下相容的錯誤修復時增加

### packages.yml 中的版本範圍

在您的 `packages.yml` 檔案中，您可以使用版本範圍來指定依賴項：

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
  - package: calogica/dbt_expectations
    version: ">=0.8.0"
```

這些範圍讓 dbt 在安裝相容版本時具有靈活性，但這種靈活性可能會導致不一致。

## 為什麼套件固定 (Package Pinning) 很重要

套件固定意味著將依賴項鎖定到特定的、確切的版本，而不是版本範圍。這對以下幾個原因至關重要：

### 可重複的構建 (Reproducible Builds)

如果沒有固定版本，在不同的機器或不同的時間執行 `dbt deps` 可能會安裝不同的套件版本，即使使用的是同一個 `packages.yml` 檔案。這可能導致：

- 模型在一台機器上運作正常，但在另一台機器上失敗
- 當套件更新時，出現意外的行為變化
- 在生產環境中出現難以除錯的問題

### 團隊一致性

當整個團隊都使用相同的固定版本時，每個人的套件依賴項都是完全相同的。這消除了困擾數據團隊的「在我的機器上運作正常」的問題。

### 受控的更新

有了固定版本，套件更新就變成了有意的決策，而不是意外的副作用。您可以在更新進入生產環境之前，先在開發環境中進行測試。

## package-lock.yml 的運作原理

當您執行 `dbt deps` 時，dbt 會：

1. 讀取您的 `packages.yml` 檔案
2. 解析所有依賴項及其版本範圍
3. 下載並安裝套件
4. 在 `package-lock.yml` 中記錄安裝的確切版本

鎖定檔案包含每個套件的詳細資訊：

```yaml
packages:
- package: dbt-labs/dbt_utils
  version: 1.1.1
  name: dbt_utils
- package: calogica/dbt_expectations  
  version: 0.10.1
  name: dbt_expectations
sha1_hash: b980d166c84935d6d661192cf31db5144295c1af
```

### 了解 SHA1 哈希值 (SHA1 Hash)

您會注意到鎖定檔案底部有一個 `sha1_hash`。這是一個唯一的指紋，代表所有依賴項組合後的精確狀態。您可以將其想像成包裹上的防篡改封條——如果您的依賴項有任何變動（版本更改、新增或移除套件），這個哈希值也會隨之改變。

SHA1 哈希值是驗證兩個環境是否具有相同依賴設置的快速方法。如果兩個團隊成員擁有相同的哈希值，dbt 就可以確信他們使用的是完全相同的套件版本，而無需逐一檢查每個套件。

在隨後執行 `dbt deps` 時，dbt 將使用鎖定檔案中指定的版本，而不是從 `packages.yml` 解析範圍，從而確保一致性。

## 我應該將 package-lock.yml 提交到版本控制嗎？

**是的，您應該將 `package-lock.yml` 提交到您的版本控制系統 (Git)。** 原因如下：

### 提交的好處

- **團隊同步**：所有團隊成員都獲得相同的套件版本
- **環境一致性**：開發、預發佈 (Staging) 和生產環境使用相同的依賴項
- **部署可靠性**：CI/CD 流程安裝可預測的套件版本
- **審計追蹤**：追蹤套件版本隨時間變化的情況

### 最佳實踐

- 將鎖定檔案與 `packages.yml` 的變更一起提交
- 在代碼審查 (Code Review) 過程中包含鎖定檔案的更新
- 在有意更新套件時重新生成鎖定檔案

## 管理套件更新

在使用鎖定檔案時更新套件的步驟：

1. 如果需要，**更新 packages.yml** 中的新版本範圍
2. **刪除 package-lock.yml** 以強制重新解析
3. **執行 dbt deps** 以生成包含更新版本的新鎖定檔案
4. **進行徹底測試** 以確保相容性
5. **同時提交這兩個檔案**


## 常見問題排除

### 鎖定檔案衝突

如果團隊成員擁有不同的鎖定檔案：
1. 刪除鎖定檔案
2. 執行 `dbt deps` 生成一個新的
3. 提交新的鎖定檔案

### 套件版本衝突

如果套件之間存在衝突的依賴項：
1. 審查您的 `packages.yml` 版本範圍
2. 考慮更新到更新的套件版本
3. 查看套件文件以獲取相容性資訊

## 結論

`package-lock.yml` 檔案是維護一致、可重複 dbt 專案的強大工具。透過自動固定您的套件版本，它消除了一個主要的環境相關問題來源，使您的 dbt 專案更加可靠。

請記住將您的鎖定檔案提交到版本控制。透過適當的鎖定檔案管理，您將花費更少的時間來除錯套件相關問題，而有更多的時間來構建有價值的數據模型。

欲了解更多關於 dbt 套件管理和最佳實踐的資訊，請查看 [dbt 官方文件](https://docs.getdbt.com/docs/build/packages)。
