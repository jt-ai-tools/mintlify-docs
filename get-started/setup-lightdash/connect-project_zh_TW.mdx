這是您想要連接 Lightdash 的 Google Cloud Platform 專案 ID。

要尋找您的專案 ID，請前往 [BigQuery Cloud 控制台](https://console.cloud.google.com/)。

進入後，點擊頂部欄位的專案名稱。

<Frame>
  <img src="/images/get-started/setup-lightdash/connect-project/get-to-projects-bd5a9153523381e91b00033480cb862c.png" alt=""/>
</Frame>

然後，您應該能在組織中看到所有您有權存取的專案及其專案 ID。

<Frame>
  <img src="/images/get-started/setup-lightdash/connect-project/project-id-list-840be6a159a39476ecf24eb7fea43a08.png" alt="" />
</Frame>

針對您要連接 Lightdash 的專案，只需複製其 `id` 並將其填入 Lightdash 數據倉庫連接表單中的 `project` 欄位即可。


##### 驗證類型

您可以選擇使用使用者帳戶（使用「透過 Google 登入」）或使用 JSON 金鑰檔案的服務帳戶來連接 BigQuery。

##### 使用者帳戶 (透過 Google 登入)

當您使用「透過 Google 登入」時，Lightdash 將使用您的個人 Google 使用者帳戶對 BigQuery 執行查詢。這是使用現有帳戶快速連接的最簡單方法。

##### 服務帳戶 (JSON 金鑰檔案)

要讓 Lightdash 透過服務帳戶連接到 BigQuery，您需要建立該帳戶和 JSON 金鑰。您可以在 [Google BigQuery 文件中閱讀更多關於建立和管理服務帳戶的資訊](https://cloud.google.com/iam/docs/creating-managing-service-accounts)。您需要具備在 Google 專案中建立服務帳戶和金鑰的權限。如果您沒有這些權限，請改用使用者帳戶。

<Info>
  如果您正在建立新的服務帳戶，請確保將 JSON 金鑰檔案儲存在安全的地方！連接 Lightdash 時會用到它。
</Info>

您在 Lightdash 中使用的服務帳戶需要在 GCP 專案中具備以下角色：

* `roles/bigquery.dataViewer`（用於查看專案中的數據）

* `roles/bigquery.jobUser`（用於在專案中執行查詢）

如果您需要提供跨多個 BigQuery 專案的數據存取權限，則需要為每個額外的 BigQuery 專案授予服務帳戶 `roles/bigquery.dataViewer` 權限。

準備好服務帳戶後，您需要在 Lightdash 數據倉庫連接頁面的 `key file` 部分添加其 JSON 金鑰檔案。

##### 位置 (Location)

這是 BigQuery 中寫入 dbt 模型輸出數據集的數據位置。

`location` 可以是多區域位置（例如 `EU`、`US`）或區域位置（例如 `us-west2`）。請參閱 [BigQuery 文件](https://cloud.google.com/bigquery/docs/locations) 以獲取有關數據集位置的更多資訊。

您可以在 [dbt profiles.yml 檔案中](https://docs.getdbt.com/reference/warehouse-profiles/bigquery-profile#dataset-locations) 或在 BigQuery 控制台中找到您 dbt 專案所使用的數據集位置。

<Frame>
  <img src="/images/get-started/setup-lightdash/connect-project/data-location-55e6d6d1001e53864486bf81014ff921.png" alt="" />
</Frame>


##### 超時秒數 (Timeout in seconds)

BigQuery 支援查詢超時。預設情況下，超時設置為 300 秒。如果 Lightdash 執行的查詢超過此超時時間仍未完成，BigQuery 可能會取消查詢並發出以下錯誤：

```bash
Operation did not complete within the designated timeout.
```

要更改此超時時間，請使用 `Timeout in seconds` 配置。


##### 優先級 (Priority)

Lightdash 執行的 BigQuery 作業優先級可以在數據倉庫連接設置中的 `priority` 配置。`priority` 欄位可以設置為 `batch`（批次）或 `interactive`（交互式）之一。

有關查詢優先級的更多資訊，請[參閱 BigQuery 文件。](https://cloud.google.com/bigquery/docs/running-queries)


##### 重試次數 (Retries)

`retries` 配置指定了 Lightdash 應重試因未處理的伺服器錯誤而導致失敗的查詢次數。

例如，將 `retries` 設置為 5 意味著 Lightdash 將延遲重試 BigQuery 查詢 5 次。如果第五次嘗試後查詢仍未成功，Lightdash 將拋出錯誤。

預設重試次數設置為 3 次。


##### 最大計費位元組數 (Maximum bytes billed)

如果設置了 `Maximum bytes billed` 的值，則由 Lightdash 執行的查詢若超過配置的最大位元組數門檻，則會失敗。此配置應以位元組數的整數形式提供。

例如，將其設置為 `1000000000` 意味著如果查詢的計費數據超過 1GB（例如 2GB），BigQuery 將拒絕該查詢，您會得到如下錯誤：

```bash
  Query exceeded limit for bytes billed: 1000000000. 2000000000 or higher required.
```


##### 每週起始日 (Start of week)

這控制了 Lightdash 中一週的起始日。`Auto` 會將其設置為數據倉庫的預設值。或者，您可以自定義並從下拉選單中選擇。這在 Lightdash 中使用「WEEK」時間間隔時會被納入考量。


##### 執行專案 (Execution project)

您可以在此指定一個執行專案來支付查詢執行費用，而不是使用實例化 dbt 資源的專案。如果您將此留空，所有成本都將應用於連接詳情頂部的專案。

### Postgres

您可以在 [dbt 文件](https://docs.getdbt.com/reference/warehouse-profiles/postgres-profile) 中查看更多詳細資訊。

這是資料庫使用者密碼。

##### 資料庫名稱 (DB name)

這是資料庫的名稱。

##### Schema

這是 dbt 用於編譯和執行 dbt 專案的預設 schema。您可以在 dbt cloud IDE 或本地的 `profiles.yml` 檔案中找到它。

如果您是 dbt cloud 使用者，可以在 dbt cloud IDE 的個人資料下找到：

<Frame>
  <img src="/images/get-started/setup-lightdash/connect-project/dbt-cloud-credentials-351bb1a9bdda693be0ed82eb17705410.png" alt="" />

如果您在本地使用 dbt，請檢查位於 `~/.dbt/profiles.yml` 的 `profiles.yml` 檔案，查找名為 `schema` 的欄位：

```yaml
company-name:
  target: dev
  outputs:
    dev:
      type: postgres
      host: [主機名稱]
      user: [使用者名稱]
      password: [密碼]
      port: [連接埠]
      dbname: [資料庫名稱]
      schema: [dbt schema] # 找這個！
```

##### 連接埠 (Port)

這是資料庫運行的連接埠。

##### 保持活動空閒秒數 (Keep alive idle (seconds))

這指定了作業系統在無網路活動多少秒後應向客戶端發送 TCP keepalive 訊息。您可以在 [postgresqlco 文件](https://postgresqlco.nf/doc/en/param/tcp%5Fkeepalives%5Fidle/) 中查看更多詳細資訊。

##### 搜尋路徑 (Search path)

這控制了 Postgres 的「搜尋路徑」。您可以在 [dbt 文件](https://docs.getdbt.com/reference/warehouse-profiles/postgres-profile#search%5Fpath) 中查看更多詳細資訊。

##### SSL 模式

這控制了 dbt 如何使用 SSL 連接到 Postgres 資料庫。您可以在 [dbt 文件](https://docs.getdbt.com/reference/warehouse-profiles/postgres-profile#sslmode) 中查看更多詳細資訊。

##### SSL 憑證 (SSL certificate)

用於驗證您與資料庫連接的客戶端憑證。僅在使用 SSL 模式 `verify-full` 時需要。

##### SSL 私鑰 (SSL private key)

用於驗證您與資料庫連接的私鑰。僅在使用 SSL 模式 `verify-full` 時需要。

##### SSL 根憑證 (SSL root certificate)

用於驗證資料庫伺服器身份的受信任憑證授權單位 (CA) 憑證。僅在使用 SSL 模式 `verify-ca` 或 `verify-full` 時需要。

##### 每週起始日 (Start of week)

這控制了 Lightdash 中一週的起始日。`Auto` 會將其設置為數據倉庫的預設值。或者，您可以自定義並從下拉選單中選擇。這在 Lightdash 中使用「WEEK」時間間隔時會被納入考量。

##### 使用 SSH 隧道 (Use SSH tunnel)

啟用後可輸入您的 **SSH 遠端主機**、**SSH 遠端連接埠**、**SSH 使用者名稱**，並生成公鑰。

##### 驗證類型

選擇使用使用者帳戶（使用「透過 Snowflake 登入」）或使用 JSON 金鑰檔案或密碼的服務帳戶進行驗證。

##### 透過 Snowflake 登入

此方法要求您從 Snowflake 數據倉庫配置 Oauth2 流程。

<Accordion title="在 Snowflake 數據倉庫上添加安全整合">

要執行此操作，首先在 Snowflake 數據倉庫上打開 SQL 控制台，並像這樣建立一個 `security integration`：

```
CREATE SECURITY INTEGRATION lightdash_oauth
  TYPE = OAUTH
  ENABLED = TRUE
  OAUTH_CLIENT = 'CUSTOM'
  OAUTH_CLIENT_TYPE = 'CONFIDENTIAL'
  OAUTH_REDIRECT_URI = 'https://<customer>.lightdash.cloud/api/v1/oauth/redirect/snowflake'
  OAUTH_ISSUE_REFRESH_TOKENS = TRUE
```

現在您可以使用以下指令獲取整合詳情：

```
DESCRIBE SECURITY INTEGRATION lightdash_oauth;
```

複製 `OAUTH_AUTHORIZATION_ENDPOINT` 和 `OAUTH_TOKEN_ENDPOINT`。

然後執行：

```
SELECT SYSTEM$SHOW_OAUTH_CLIENT_SECRETS('LIGHTDASH_OAUTH');
```

並複製 `OAUTH_CLIENT_SECRET`、`OAUTH_CLIENT_SECRET_2` 和 `OAUTH_CLIENT_ID`。

最後，聯繫 Lightdash 團隊 support@lightdash.com，為您的實例設置「透過 Snowflake 登入」。

</Accordion>

您可以在 [Snowflake 官方文件](https://docs.snowflake.com/en/user-guide/oauth-snowflake-overview) 中閱讀更多內容。

##### 私鑰 (Private Key)

您可以按照 [此處的指南](https://docs.snowflake.com/en/user-guide/key-pair-auth#generate-the-private-key) 為 Snowflake 使用者生成私鑰。

生成後，將生成的私鑰檔案內容複製到 Lightdash 中。如果您選擇加密私鑰，還需要提供私鑰密碼 (Passphrase)。

##### 密碼 (Password)

這是您 Snowflake 使用者的密碼。這通常與您登入 Snowflake 時使用的密碼相同。請注意，由於 Snowflake 驗證機制的更改，需要密碼的使用者可能還需要啟用 MFA，而 MFA 與 Lightdash 專案連接不相容。

##### 角色 (Role)

這是您在以指定使用者身份執行查詢時想要使用的安全角色。該角色必須具備存取您想使用的任何數據倉庫、資料庫、schema 和表格的權限。

<Accordion title="我正在使用 dbt Cloud，我應該透過 git 儲存庫還是透過 dbt Cloud 連接？">

如果您透過 git 儲存庫連接，需要確保 dbt Cloud 中的環境版本設置為 [「Compatible」(相容)](https://docs.getdbt.com/docs/dbt-versions/cloud-release-tracks)。

</Accordion>

我們建議您使用 OAuth 連接到 GitHub。這為 Lightdash 提供了與儲存庫的直接連接，因此當個人離職時不會失去存取權限，並且可以建立拉取請求 (Pull Request)，這是 [dbt 寫回功能](/guides/developer/dbt-write-back_zh_TW) 所必需的。此連接存在於組織層級，因此您可以將不同的 dbt 專案儲存庫映射到每個 Lightdash 專案，而無需新的存取權杖。

### Bitbucket

##### 使用者名稱 (Username)

這是您 Bitbucket 使用者的登入名稱。這通常與您登入 Bitbucket 時使用的使用者名稱相同。您可以按以下步驟在 Bitbucket 中找到您的使用者名稱：

1. 點擊左下角的頭像，然後點擊 **Personal settings**。

2. 在 **Account settings** 頁面上，查看 **Username** 旁邊的值。

或者，您可以[透過 Bitbucket 控制台建立新使用者](https://confluence.atlassian.com/bitbucketserver/users-and-groups-776640439.html)，專門為 Lightdash 提供一個使用者名稱和密碼。

##### HTTP 存取權杖 (Http access token)

獲取權杖的方式取決於您使用的是 Bitbucket Cloud 還是 Bitbucket Server：

* Bitbucket Cloud 使用者應[參考建立 App Password 的說明](https://support.atlassian.com/bitbucket-cloud/docs/create-an-app-password/)

* Bitbucket Server 使用者應[參考建立 HTTP Access Token 的說明](https://confluence.atlassian.com/bitbucketserver/http-access-tokens-939515499.html)

建立權杖時選擇 `Project read` 和 `Repository read` 範圍。

##### 儲存庫 (Repository)

格式應為 `my-org/my-repo`。例如：`lightdash/lightdash-analytics`

##### 分支 (Branch)

這是您 Bitbucket 儲存庫中 Lightdash 應同步的分支。例如：`main`、`master` 或 `dev`

預設情況下，我們將其設置為 `main`，但您可以根據需要進行更改。

##### 專案目錄路徑 (Project directory path)

這是您在上述 Bitbucket 儲存庫中找到 `dbt_project.yml` 檔案的資料夾。

* 如果您的 `dbt_project.yml` 檔案位於儲存庫的根目錄中，請填入 `/`（例如：lightdash/lightdash-analytics/dbt\_project.yml）

* 如果您的 dbt 專案在子目錄中，請包含子目錄的路徑。例如，如果我的專案在 lightdash/lightdash-analytics/dbt/dbt\_project.yml，我會在此欄位中填入 `/dbt`。

##### 主機網域 (Host domain)

如果您[為 Bitbucket Server 自定義了主機網域](https://confluence.atlassian.com/bitbucketserver/specify-the-bitbucket-base-url-776640392.html)，可以在此處填入。

### dbt Cloud

dbt Cloud 連接類型繞過了任何 git 連接，而是直接使用 dbt Cloud API 存取 dbt Cloud 專案的元數據。

這意味著許多功能**不可用**，包括：
* [dbt 寫回功能](/guides/developer/dbt-write-back_zh_TW)
* 從 git 分支在 UI 中建立的[預覽環境 (Preview environments)](/references/preview-projects_zh_TW#lightdash-app)
* 在 git 儲存庫有新提交時自動刷新——當您更改 dbt 專案時，您需要在 Lightdash 中手動觸發刷新。

##### 服務權杖 (Service Token)

要將 Lightdash 連接到您的 dbt Cloud 專案，您需要建立一個具備「Metadata Only」權限的 [dbt Cloud 服務權杖](https://docs.getdbt.com/docs/dbt-cloud-apis/service-tokens)。

##### 環境 ID (Environment ID)

這是您 dbt Cloud 環境的識別碼。您可以在 dbt Cloud 環境的 URL 中找到它。Lightdash 將使用此 ID 來識別應從哪個環境構建，因此通常您會指向您的生產環境。

##### dbt Webhook

此 Webhook 對於使用 dbt Cloud CI 步驟的使用者來說是一個選填項。如果您設置了在 dbt Cloud CI 作業完成時觸發的 Webhook，您可以貼上 Webhook 位址，Lightdash 將自動為該 CI 環境啟動預覽。

##### Discovery API 端點 (Discovery API Endpoint)

適用於您的 dbt Cloud 執行體的 dbt 特定端點。您可以在 [dbt Cloud API 文件](https://docs.getdbt.com/docs/dbt-cloud-apis/discovery-querying#discovery-api-endpoints) 中找到詳細資訊。

### 本地 dbt 專案 (Local dbt project)

<Warning>
  不適用於生產環境，且僅適用於安裝在您本地機器上的 Lightdash 實例。
</Warning>

要在啟動 Lightdash 時選擇連接到本地 dbt 專案，您必須在啟動 docker compose 時指定 dbt 專案的目錄：

```yaml
# 指定 dbt 專案的絕對路徑
# 例如：export DBT_PROJECT_DIR=/Users/jake/mydbtproject
export DBT_PROJECT_DIR= # 在此填入您的路徑！
docker compose start
```
